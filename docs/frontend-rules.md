# Frontend Rules

Этот файл хранит текущие договоренности по архитектуре и стилю проекта.
Если правило меняется, обновляем сначала этот документ, потом код.

## Layers

- `shared`: только общий код без привязки к конкретному бизнес-домену.
- `entities`: API, схемы, типы, query hooks конкретной сущности.
- `features`: пользовательские сценарии поверх сущностей.
- `widgets`: композиция нескольких features/entities/shared.
- `pages`: сборка страницы из widget.
- `app`: bootstrap, router, providers, theme, app-level orchestration.
- `feature` не импортирует `feature`.
- `entity` не импортирует `entity`.
- `widget` не импортирует `widget`.
- `page` не импортирует `page`.
- Если код нужен нескольким слайсам одного слоя, его опускаем ниже в `shared` или поднимаем выше в слой-компоновщик.

## Exports

- Barrel exports оставляем.
- `index.ts` должен экспортировать только внешний public API слоя.
- Если сущность, хук, helper, тип или UI используются только внутри своей папки, наружу их не экспортируем.
- Не делаем `export *` "на всякий случай" из файлов, которые содержат внутренние детали.

## Schemas

- Если схема не зависит от локализации, она должна быть обычной константой.
- Если схема использует локализованные тексты ошибок, она должна быть schema factory с аргументом `t`.
- Не использовать глобальный `i18next.t(...)` внутри form schema, если можно передать `t` явно.
- Response/payload schemas остаются в `schemas.ts`.

## Types

- Type aliases храним в `types.ts`.
- Типы выводим из схем через `z.infer`.
- Не дублируем shape вручную, если уже есть zod schema.
- Если тип нужен только внутри соседнего `api.ts`, не обязательно экспортировать его из `types.ts`; можно вывести локально в самом `api.ts`.
- Не держим в `types.ts` промежуточные helper-типы, если они нигде не используются.

## Imports

- Внутри одного и того же слоя/слайса используем относительные импорты.
- Alias-импорты используем только для импорта из других слоев или из внешнего public API.
- Не делать self-import через barrel своего же слоя.

## Features And UI

- Если компонент используется только внутри одной feature, он живет в этой feature.
- В `shared/ui` выносим только реально переиспользуемые компоненты.
- Тяжелый JSX из `columns` и похожих конфигов выносим в отдельные UI-компоненты, если он начинает разрастаться.

## Helpers

- Helper-функции не держим вперемешку в `columns`, `widget`, `page` и подобных файлах, если они начинают жить своей жизнью.
- Если helper нужен только внутри одной feature/entity, выносим его в локальный `helpers`.
- Если helper реально переиспользуется между несколькими слайсами, выносим его в `shared/helpers`.
- Не экспортируем helper наружу, если он нужен только внутри своей папки.

## Forms

- Формы собираем на `react-hook-form`.
- Валидацию форм делаем через `zodResolver`.
- Ошибки валидации должны быть локализованы.

## API And Errors

- Для API используем entity-level functions.
- Ошибки от бэка показываем через общий helper извлечения message.
- Pending/success тексты для mutation задаем с фронта через i18n.

## When In Doubt

- Предпочитать более узкий public API.
- Предпочитать явную зависимость неявной.
- Предпочитать простую схему без лишних промежуточных слоев.
